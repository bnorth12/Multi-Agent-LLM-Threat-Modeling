# Project Requirements

|ID|Name|Requirement Text|Requirement Rationale|Verification Method|Verification Statement|
|---|---|---|---|---|---|
|PRJ-001|Unified Input Ingestion|Threat Modeler SHALL ingest free-text architecture descriptions and tabular data-flow inputs in a single analysis run.|Unified ingestion is required for practical analyst workflows.|Demonstration|Verified by executing an end-to-end run with text and table inputs and confirming both sources are present in run artifacts.|
|PRJ-002|Canonical Graph Authority|Threat Modeler SHALL produce a canonical threat-model graph as the authoritative run artifact.|A single source of truth prevents cross-artifact drift.|Inspection|Verified by artifact review showing all downstream outputs reference canonical graph IDs.|
|PRJ-003|Deterministic Pipeline|Threat Modeler SHALL execute a deterministic multi-agent pipeline with auditable stage boundaries.|Determinism is required for repeatability and accreditation.|Test|Verified by repeated identical runs producing equivalent stage outputs and transition logs.|
|PRJ-004|Stage Validation Gate|Threat Modeler SHALL validate every stage output against a defined schema before downstream processing.|Validation prevents error propagation.|Test|Verified by injecting malformed stage output and confirming downstream stages are blocked.|
|PRJ-005|Full Threat Workflow|Threat Modeler SHALL perform trust-boundary analysis, STRIDE scoring, threat generation, mitigation mapping, STIX packaging, diagram generation, and report generation for each completed run.|The workflow defines project mission scope.|Demonstration|Verified by completed run containing all required stage outputs.|
|PRJ-006|HITL Governance|Threat Modeler SHALL support human-in-the-loop approval, edit, and override gates at defined decision points.|Analyst control is required for high-consequence systems.|Demonstration|Verified by gate workflow walkthrough showing approve, edit, and override actions.|
|PRJ-007|Immutable Auditability|Threat Modeler SHALL maintain immutable run history with who, what, when, and why metadata for all analyst edits.|Compliance and forensics require immutable provenance.|Inspection|Verified by review of audit logs containing actor, timestamp, rationale, and before-and-after diff.|
|PRJ-008|Configurable Model Selection|Threat Modeler SHALL support configurable model-provider selection through configuration without source-code changes.|Provider agility reduces integration risk.|Test|Verified by switching provider configuration and running without code modification.|
|PRJ-009|Deployment Mode Flexibility|Threat Modeler SHALL allow deployment in offline-only mode and policy-approved hybrid mode.|Deployment constraints vary by program classification.|Analysis|Verified by configuration and architecture review showing both modes and policy controls.|
|PRJ-010|Evidence-Linked Outputs|Threat Modeler SHALL provide retrieval-augmented evidence references for generated threats and mitigations when knowledge bases are enabled.|Traceable evidence improves analyst trust and review speed.|Test|Verified by threat and mitigation records including source references when retrieval is enabled.|
|PRJ-011|Export Artifact Set|Threat Modeler SHALL produce exportable outputs for canonical JSON, STIX 2.1, Mermaid diagrams, and final markdown report.|Output interoperability is required for downstream tooling.|Demonstration|Verified by successful export of all four artifact types in one run.|
|PRJ-012|Role-Based Access Control|Threat Modeler SHALL enforce role-based permissions for author, reviewer, and approver actions.|Controlled actions are required for governance and separation of duty.|Test|Verified by permission tests confirming blocked unauthorized actions and allowed authorized actions.|
|PRJ-013|Incremental Enrichment|Threat Modeler SHALL support incremental model enrichment without destructive overwrite of previously approved data.|Programs evolve and require safe incremental updates.|Test|Verified by merge run preserving approved prior elements while adding new scope.|
|PRJ-014|Selective Re-Run|Threat Modeler SHALL provide selective re-run from an analyst-selected stage after approved edits.|Selective re-run reduces cycle time.|Demonstration|Verified by restarting from selected stage and confirming prior unchanged stages are not recomputed.|
|PRJ-015|Fail-Safe Halting|Threat Modeler SHALL fail safely by halting downstream execution when critical validation errors are detected.|Safe failure prevents invalid outputs from release.|Test|Verified by induced critical validation failure causing controlled stop with error report.|
