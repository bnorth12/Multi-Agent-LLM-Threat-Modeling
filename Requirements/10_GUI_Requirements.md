# GUI Requirements

Date: 2026-05-03
Status: Draft v0.1

**HMI Architecture Blueprint:** [docs/HMI_Architecture_Blueprint.md](../docs/HMI_Architecture_Blueprint.md) is the design authority document that implements these requirements. Refer to it for navigation model, shared component patterns, screen wireframes, role-based access gating, and state management. S05-04 HITL gate implementation SHALL conform to the blueprint.

Purpose:

- Define graphical user interface requirements for all analyst-facing interactions
- Cover input entry, HITL gate screens, pipeline status, stage results, export, snapshot management, and agent prompt configuration
- Establish GUI access control alignment with system authorization policy

Planned sprint: future sprint (post Sprint 2026-05)

Related requirements:

- PRJ-006, PRJ-011, PRJ-012, PRJ-014, PRJ-016, PRJ-017, PRJ-018
- HITL-001 through HITL-011
- INT-006, INT-007, INT-008, INT-009, INT-010, INT-011, INT-013

|ID|Name|Requirement Text|Requirement Rationale|Verification Method|Verification Statement|
|---|---|---|---|---|---|
|GUI-001|Input Entry Form|The GUI SHALL provide a form-based input screen for analyst entry of system name, description, data flows, assets, trust boundaries, and deployment context prior to pipeline execution.|A structured entry form reduces input errors and ensures all required fields are captured before analysis begins.|Demonstration|Verified by walkthrough confirming all required input fields are present and submitted payload passes schema validation.|
|GUI-002|HITL Gate Screens|The GUI SHALL provide a dedicated screen for each HITL gate that presents the relevant stage outputs, supports approve, edit, and override actions, and captures rationale for all non-approve decisions.|Gate screens must give analysts full context and decision controls to meet HITL governance requirements defined in HITL-001 through HITL-011.|Demonstration|Verified by walkthrough of each defined gate screen confirming context display and all required action controls are present.|
|GUI-003|Pipeline Status Dashboard|The GUI SHALL provide a pipeline status view that displays the current executing stage, completed stages, pending stages, and any blocking conditions in real time during execution.|Analysts need situational awareness of pipeline progress to manage review timing and escalate blockers promptly.|Demonstration|Verified by executing a full run and confirming the status view updates at each stage transition.|
|GUI-004|Stage Results Viewer|The GUI SHALL provide a results viewer for each pipeline stage that renders the stage output artifacts and makes them accessible to the analyst before and after HITL review.|Per-stage visibility is required for analyst calibration, auditing, and review between gates.|Demonstration|Verified by navigating to each stage result view during and after a run and confirming stage output is rendered correctly.|
|GUI-005|Threat and Mitigation Review Screen|The GUI SHALL provide a combined threat and mitigation review screen that renders scored threats alongside their mapped mitigations and supports inline approval and edit actions per threat record.|Threat and mitigation review is a primary analyst workflow and requires consolidated context to support efficient and accurate decisions.|Demonstration|Verified by review screen walkthrough confirming threat scores, mitigation mappings, and inline edit controls are all present and functional.|
|GUI-006|Results Export Interface|The GUI SHALL provide an export panel that allows analysts to download completed run outputs in canonical JSON, STIX 2.1 bundle, Mermaid diagram, and Markdown report formats.|Exportable outputs in standard formats are required for downstream tooling integration and program deliverable packaging.|Demonstration|Verified by export panel walkthrough confirming all four format downloads are available, triggered successfully, and produce valid output files.|
|GUI-007|Run Snapshot Export|The GUI SHALL provide a snapshot export action that captures the complete run state, including inputs, all intermediate stage results, HITL decisions, prompt configuration, and run metadata, into a single portable file.|Snapshot export enables run archival, audit evidence packaging, session handoff, and future restore without loss of intermediate state.|Test|Verified by exporting a snapshot and confirming the snapshot file contains all required run state fields as defined by the snapshot schema.|
|GUI-008|Run Snapshot Restore|The GUI SHALL provide a snapshot restore action that imports a previously exported snapshot file and reconstitutes the run state so that the analyst can continue analysis, review history, or audit the prior run.|Restore capability supports program continuity, analyst session handoffs, and compliance evidence review across sessions.|Demonstration|Verified by restoring a snapshot into a fresh session and confirming the reconstituted run state matches the original exported state.|
|GUI-009|Agent Prompt Editor|The GUI SHALL provide an agent configuration screen that displays the current active prompt for each pipeline agent and allows authorized users to edit and save updated prompts without requiring source code changes.|In-GUI prompt editing enables mission tuning, policy updates, and operational adaptation by authorized non-developer roles and reduces deployment friction.|Demonstration|Verified by editing a prompt through the GUI and confirming that the next pipeline execution uses the updated prompt text.|
|GUI-010|Agent Prompt Version History|The GUI SHALL maintain a version history of saved agent prompts and allow authorized users to view prior prompt versions and revert to a selected prior version.|Version history prevents permanent prompt loss and enables rollback when prompt changes produce undesired or degraded behavior.|Inspection|Verified by saving multiple prompt versions for an agent, viewing the version list, and confirming the revert action restores the selected prior prompt in the next execution.|
|GUI-011|Role-Enforced GUI Access Control|The GUI SHALL enforce role-based access controls consistent with INT-013 such that edit, approve, override, export, snapshot, and agent prompt-edit actions are restricted to authorized roles and unauthorized actions are blocked at the GUI layer.|GUI access control must align with the system authorization policy defined in PRJ-012 to prevent unauthorized analyst actions through the interface.|Test|Verified by role matrix tests confirming that actions requiring elevated roles are blocked for unauthorized roles and permitted for authorized roles at the GUI layer.|
|GUI-012|Model Provider Selection Screen|The GUI SHALL provide a model configuration screen that displays available model providers (local, hosted cloud services, commercial API services), allows analysts to select a provider, and presents provider-specific connection options without requiring source code changes.|Analyst-driven model provider selection enables runtime flexibility to switch between local LLMs, cloud-hosted instances, and commercial services based on mission constraints and availability.|Demonstration|Verified by walkthrough confirming all available provider options are displayed, selection is retained, and the next pipeline execution targets the selected provider.|
|GUI-013|Model Connection Details Configuration|The GUI SHALL provide a configuration form for entering and editing model connection details including API endpoint, API key, authentication credentials, model name, deployment ID, and region-specific settings, with secure storage of sensitive credentials (encrypted or credential manager integration).|Connection detail configuration enables seamless integration with diverse model instances without code modification or manual config file editing.|Demonstration|Verified by entering connection details for a hosted LLM provider through the form, confirming data is persisted, and confirming the next run uses the configured endpoint and credentials.|
|GUI-014|Model Connection Validation|The GUI SHALL provide a test-connection action that validates model connectivity and authentication before commencing a pipeline run, returning a user-readable status message and blocking run initiation if validation fails.|Pre-execution validation prevents pipeline failures due to misconfigured endpoints or expired credentials and provides immediate feedback for configuration troubleshooting.|Test|Verified by test-connection action with valid and invalid credentials, confirming success message on valid connection, clear error message on invalid, and run-blocking behavior on failed validation.|
